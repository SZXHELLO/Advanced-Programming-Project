1. 项目概述
--------------------------------------------------------------------------------
1.1 产品定位
  为 minibot 增加一个 Web 前端（浏览器内使用的控制面板），侧重两类场景：
  - Dashboard：与主 Agent 对话、查看与管理多会话历史、调整模型/API 连接参数。
  - AgentsBoard：管理持久化子 Agent（subagent）——查看、创建、编辑 task、删除；数据落在工作区内的 JSON 文件中，与 CLI /addagent 等机制对齐。

1.2 技术栈（建议，可由实现 Agent 微调但需在本文件后续版本记录变更）
  - 前端：任选成熟方案（React / Vue / Svelte 等）+ 构建工具（Vite 等）。
  - 与本机 minibot 通信：WebSocket（对话）+ 本地 HTTP 代理或「小型后端」读写 JSON 配置/会话文件（见 §7）。
  - 目标平台：优先桌面浏览器本机访问（localhost）；不强制首版支持移动端适配。

1.3 非目标（首版可不实现）
  - 多用户鉴权、公网部署、端到端加密会话列表。
  - 与 QQ/Telegram 等通道 UI 深度整合（仅可读取同一 workspace 下的会话文件时顺带展示）。

--------------------------------------------------------------------------------
2. 与现有 minibot 集成（必读）
--------------------------------------------------------------------------------
2.1 运行前置
  - 用户需已安装本仓库 minibot，并能成功执行 `minibot gateway`（或等价启动方式）。
  - WebSocket 通道需在配置中启用，参见仓库文档 `docs/WEBSOCKET.md`。典型配置片段：
    `channels.websocket.enabled: true`，`host` / `port` / `path` / `streaming` 等。

2.2 对话传输（Dashboard）
  - 协议：JSON 文本帧，事件字段 `event`；连接后服务端先发 `ready`（含 `chat_id`、`client_id`）。
  - 客户端发送：`{"content": "..."}` 或 `{"text": "..."}` 等（见 WEBSOCKET.md「Client → Server」）。
  - 服务端下发：`message`（完整回复）、`delta` / `stream_end`（流式，当 streaming 开启时）。
  - 每个 WebSocket 连接会分配新的 `chat_id`（UUID）。同一「聊天会话」若要复用后端上下文，须重复使用同一连接或在产品层绑定同一 `chat_id` 的会话文件（见 §6.2）。

2.3 子 Agent 数据（AgentsBoard）
  - 权威存储：工作区目录下 `{workspace}/.minibot/persistent_subagents.json`。
  - 文件结构顶层为：`{"version": 1, "records": [ ... ]}`。**不得**使用不存在的文件名（如 `subagent_tasks.json`）。
  - 每条 record 为对象，与运行时一致的核心字段包括（节选，以代码为准）：
    `id`（字符串，唯一）、`label`、`task`、`session_key`、`status`（如 standby / running / interrupted / completed 等）、
    可能还有 `origin_channel`、`origin_chat_id` 等。
  - 创建新 subagent 时：必须生成新的唯一 `id`（UUID 无连字符或标准 UUID 均可，与现有测试/实现一致即可），并写入 `records`；**原子写入**推荐参考 Python 侧 `SubagentPersistence._atomic_write` 的语义（先写临时文件再 replace），前端若通过本地服务写文件也应避免半写入状态。

3. 信息架构与导航
--------------------------------------------------------------------------------
3.1 两个主界面
  - 「Dashboard」与「AgentsBoard」为平级入口：顶部导航或侧栏切换 Tab / 路由（路径命名可由实现定，例如 `/` 与 `/agents`）。

3.2 通用
  - 清晰指示当前 minibot 是否已连接（WebSocket 连通性、最后一次心跳或错误提示）。
  - 面向本机使用：错误信息需可读（连接拒绝、CORS、文件不可写等）。

--------------------------------------------------------------------------------
4. 界面规格 — Dashboard（对话）
--------------------------------------------------------------------------------
4.1 布局
  - 主区域：聊天窗口（消息气泡或同等展示），支持用户消息与 Agent 回复区分。
  - 若开启流式：按 `delta` 增量追加，`stream_end` 结束一段；最终以 `message` 或流式拼接结果为准（与 WEBSOCKET.md 一致）。
  - 左侧：可折叠「会话历史」侧栏（抽屉式亦可）。

4.2 会话历史侧栏
  - 列表项：每次会话一条；**显示该会话「最后活动时间」**，格式：**年-月-日 时:分**（24 小时或用户本地习惯，首尾一致即可）。
  - 排序：**最近活动的会话排在最上**（降序按 `updated_at` 或等价时间戳）。
  - 交互：
    - 点击某条：切换到该会话上下文（见 §6.2 如何映射到后端会话）。
    - 「新对话」：开启新会话（新 WebSocket 连接或新 chat_id 策略，见 §7）。
    - 「删除」某条历史：**从 UI 列表移除**，并删除磁盘上对应会话存储（见 §6.2），需确认对话框以防误删。

4.3 顶栏 / 工具
  - 右上角「Model」按钮：打开模态对话框，字段至少包括：
    - `api_key`
    - `base_url`（OpenAI 兼容 API 根地址）
    - `model`（模型名称字符串，与 minibot `agents.defaults.model` 语义一致）
  - 点击「确定」：**持久化到本机 minibot 配置文件**，并使后续对话生效（见 §6.1）。
  - 可选：展示当前已保存的配置摘要（脱敏后的 key）。

--------------------------------------------------------------------------------
5. 界面规格 — AgentsBoard（子 Agent）
--------------------------------------------------------------------------------
5.1 数据加载
  - 从 `{workspace}/.minibot/persistent_subagents.json` 读取 `records`；解析失败时给出错误提示而非白屏。
  - 每个子 Agent 以**卡片/方框**展示，至少显示：`label`、`task`（可截断长文本，点击进入详情再全显）。

5.2 点击卡片
  - 中央模态：可编辑 `task`（多行文本）；可选展示只读字段 `id`、`status`、`session_key`（便于排错）。
  - 提供「保存」：写回 JSON 中对应 `id` 的 record（保持其他字段合理合并，避免覆盖未知字段为 null）。
  - 提供「删除」：从 `records` 移除该 `id`；若 minibot 进程正在运行，删除仅影响持久化文件——可附加提示「运行中任务可能不会自动停止，需用户自行处理」（与首版非目标一致，但文案需诚实）。

5.3 创建新 Agent
  - 「创建」按钮 → 模态：输入 `label`、`task`；确定后追加新 record（含新 `id`、合理默认值如 `status: "standby"`、`session_key` 需与 CLI/Web 场景对齐——建议默认 `websocket:<chat_id>` 或 `web:ui` 等新 session_key 约定，**须在实现时写死单一默认并在 README 说明**，以便与主会话工具路由一致；若不确定，优先沿用主界面当前 effective session_key，见 §7）。

6. 数据契约与路径解析
--------------------------------------------------------------------------------
6.1 模型与 Provider（config.json）
  - 用户配置文件默认路径：`~/.minibot/config.json`（Windows 为 `%USERPROFILE%\.minibot\config.json`）。
  - Python API：`minibot.config.loader.get_config_path()`、`load_config` / `save_config`。
  - UI 写入建议字段（与 OpenAI 兼容自定义网关常见用法）：
    - `agents.defaults.model` ← 对话框 `model`
    - `providers.custom.apiKey` 或 `api_key`（配置为 camelCase / snake_case 双接受，以 schema 为准）← `api_key`
    - `providers.custom.apiBase` / `api_base` ← `base_url`
    - 若用户希望指定单一 provider 且非 auto：`agents.defaults.provider` 可设为 `custom`（需与现有 schema 允许值一致，实现前 grep `ProviderConfig` / registry）
  - **重要**：保存配置前建议做 JSON 备份；写入后应提示用户「部分场景需重启 gateway」——若代码热加载未覆盖 provider，以实测为准并在帮助文档写明。

6.2 会话历史（sessions 目录）
  - 工作区路径来自配置：`agents.defaults.workspace` 展开后路径（如 `~/.minibot/workspace`）。
  - 会话文件由 `SessionManager` 维护：目录为 `{workspace}/sessions/`，文件名与 `session.key` 相关（安全文件名变换，冒号等会替换）。
  - 会话 metadata（含 `created_at`、`updated_at`）存于 JSONL/JSON 中，具体格式以 `minibot/session/manager.py` 为准；侧栏时间与排序应基于磁盘文件中的 `updated_at` 或最后一条消息时间戳。
  - **WebSocket 会话 key** 形态通常为 `websocket:{chat_id}`（实现者须在代码中核对 `InboundMessage.session_key` 的构造处）。Dashboard「会话列表」应对 `websocket_*` 文件解析并与侧栏项绑定。
  - 删除会话：删除对应 `sessions` 下文件（含 `.jsonl` 与 `.json` 若存在），与 SessionManager 行为保持一致，避免残留。

6.3 子 Agent 存储路径
  - `{workspace}/.minibot/persistent_subagents.json`，其中 `workspace = Path(agents.defaults.workspace).expanduser()`。

7. 后端缺口与推荐架构（Agent 必须做决策并落地）
--------------------------------------------------------------------------------
现状：minibot 核心已提供 WebSocket 对话与磁盘持久化，但**通常未提供**统一的 REST API 用于「列会话」「删会话」「改 config.json」「改 persistent_subagents.json」。前端仅凭浏览器无法安全任意读写用户磁盘，因此下列三选一（或组合）：

方案 A — 本地 BFF（推荐首版）
  - 在本仓库或前端 monorepo 中增加极小 **Node/Python** 服务（localhost-only），暴露：
    - `GET/POST /api/config`（读写 config.json）
    - `GET/DELETE /api/sessions`、`GET /api/sessions/:id/messages`（封装 SessionManager 或直读文件）
    - `GET/PATCH/POST/DELETE /api/subagents`（封装 SubagentPersistence 语义）
  - Web 前端只请求 `127.0.0.1`，CORS 允许本地前端端口。
  - **优点**：权限清晰、易测；**缺点**：多一个进程，需在文档写出启动命令。

方案 B — 扩展 minibot gateway
  - 在 `minibot` 的 gateway HTTP 服务中增加路由（若已有 `GatewayConfig` 端口可挂载）。需 Python 实现并随 `minibot gateway` 启动。
  - **优点**：单进程；**缺点**：改动核心，-review 成本高。

方案 C — 桌面壳
  - Electron/Tauri 主进程读文件 + 渲染 WebView，无额外 BFF。
  - **优点**：无 CORS；**缺点**：打包与分发更重。

**会话与 WebSocket 的绑定策略（须在实现说明中选定一种并写 README）**
  - 策略 1：每个侧栏会话对应固定历史 `chat_id`，打开会话时**新建** WebSocket 并在首条消息带「恢复会话」——若协议不支持，则需扩展服务端或仅前端载入历史只读（无法真正续写同一 agent 上下文）。若不能续写，必须在 UI 标明「仅查看历史」。
  - 策略 2（简版首版）：侧栏「会话」= 已完成对话的归档视图；新消息始终新建连接，会话文件由后端照旧生成；用户通过列表打开历史文件只读。此策略实现快，但「切换会话续聊」能力弱。
  - 策略 3：优先实现 BFF 调用内部 API 若未来 minibot 暴露「指定 chat_id 连接」——当前默认 WebSocket 每次新 UUID，若需续聊必须由核心支持或策略 2。

实现 Agent 应在 `web/README.md`（或项目内等价文档）中写明：**选定方案 A/B/C** 与 **会话策略 1/2/3**，避免后续维护者误以为「侧栏切换已自动重连同一上下文」。

8. 质量与安全
--------------------------------------------------------------------------------
8.1 安全
  - 任何本地 API 必须 **bind 127.0.0.1**，禁止默认 0.0.0.0。
  - `api_key` 在界面默认掩码显示；日志不得打印完整密钥。
  - 对 subagent JSON 与 config 的写入做并发保护（文件锁或串行队列）。

8.2 可访问性 / UX
  - 模态可键盘关闭（Esc）、焦点陷阱合理。
  - 长按/二次确认删除会话与删除 subagent。

8.3 测试
  - 前端：关键组件单元测试；WebSocket 客户端 mock 测试。
  - 集成：启动 minibot + BFF（若有），跑一条发送与流式接收用例。
  - 与 Python 对齐：可对 `persistent_subagents.json` 样板做 golden 文件测试。
